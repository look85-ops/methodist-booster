## Инструкция по продолжению разработки контента для курса «Профессионал +. Путь» (Версия 2.3)

Данный документ содержит зафиксированный контекст проекта, стилистические и методические настройки, шаблоны запросов и требования к результату. Актуализирован по итогам полной разработки Модуля 1 и интеграции процедур метрик/A/B. Следуйте ему при работе над Модулем 2 и последующими.

Примечание по версии 2.3:
- Добавлена секция про метрики, A/B и привязку к АОС (п.8).
- Уточнены требования к оформлению результатов (тайминг без пересечений, метки A/B, рефлексия «что сделаю завтра»).

---

### 1. КОНТЕКСТ ПРОЕКТА (Не подлежит изменению)

Название программы: «Профессионал +. Путь»

Тип: Очный семинар‑практикум, повышение квалификации.

Целевая аудитория:
- Начальники дистанции пути (ПЧ).
- Заместители начальников дистанций пути и инфраструктуры.
- Главные инженеры дистанций пути и инфраструктуры.

Особенности аудитории (Критически важно):
1. Скепсис и сопротивление: обучение воспринимается как навязанное. Частые возражения: «всё равно мы ничего не изменим», «я ни на что не влияю, это система», «у меня нет времени».
2. Разделение «Мы — Они»: аудитория ментально отделяет себя от управляющих структур («они», «у них»).
3. Ценностная задача курса: мягко, без пропаганды, сместить фокус с «они должны» на «мы можем влиять на процессы в рамках своей зоны ответственности».

Цель всей программы:
Повторить основные нормативные положения, актуализировать и восстановить прикладные инженерные знания по базовым железнодорожным тематикам и переосмыслить их с учётом накопленного практического опыта.

Формат работы: микс теории и практики, решение задач из повседневной деятельности.

---

### 2. РОЛИ И ТОН КОММУНИКАЦИИ (Настройки «Температуры»)

Роль преподавателя: модератор‑практик, а не лектор‑транслятор нормативки.

Тон общения:
- Партнёрский, уважительный к опыту аудитории.
- Без менторской позиции «сверху».
- Активное использование профессиональной лексики («окно», приведённая длина, неудовлетворительный километр, «залежные» ТМЦ, выплеск, куст негодных шпал, очаговый перегон).

Лингвистические триггеры:
- Вместо «Вы должны знать...» → «Нам важно соотнести наш опыт с тем, как это регулируется...»
- Вместо «РЖД требует...» → «Согласно действующему регламенту...»
- Вместо «У них новая концепция...» → «Давайте посмотрим на вектор развития и оценим, какие инструменты мы можем применить на своих участках».
- Присвоение: «Ваша дистанция», «Наша общая задача», «Ваша зона ответственности».

Приёмы работы со скепсисом:
- Провокационные вопросы: «Почему при выполнении плана премия может быть меньше?», «Что реально мешает механизации?»
- Кейсы‑симуляторы: постановка в позицию ЛПР при ограничениях (защита бюджета, приоритизация ремонтов).
- Возврат к личному опыту: запрос реальных ситуаций и их анализ в конце блока.
- Экономический перевод: конвертация абстрактных понятий безопасности в конкретные цифры потерь (поездо‑часы × стоимость).

---

### 3. МЕТОДИЧЕСКАЯ СТРУКТУРА ЗАНЯТИЯ

Каждый логический блок строится по схеме «Вызов — Осмысление — Присвоение»:

1. Вызов (интерактивный вход): вопрос/ситуация, опирающаяся на опыт аудитории. Фиксация мнений на флипчарте. Допустим «шок‑контент» (примеры крушений с переводом в экономику).
2. Осмысление (мини‑лекция / норматив): подача теории тезисно, строго по материалам. Визуализация без «простыней» текста. Обязательная связка с реальностью.
3. Присвоение (практикум / дискуссия): применение знания к своему участку. Групповая работа, защита решений. Обязательная рефлексия: «Что было трудно?», «Какое одно улучшение внедрите завтра?»

---

### 4. ОПЫТ МОДУЛЯ 1: ФОРМАТ «ЕДИНОГО КЕЙСА» ДНЯ

Для заключительных дней модуля (формат «Хакатон») применять Вариант 1 «Лёгкая сюжетная рамка»:
- Суть: все задания объединяются сквозной легендой (например, «вы — команда N‑ской дистанции»). Условия заданий связываются сюжетными вставками (письма, приказы, планёрки).
- Преимущества: целостное восприятие дня, погружение в управленческий цикл, снижение когнитивной нагрузки.
- Реализация: добавление связующих слайдов между этапами и лёгкая корректировка вступительных текстов преподавателя.

---

### 5. ТРЕБОВАНИЯ К ОФОРМЛЕНИЮ РЕЗУЛЬТАТОВ

- Для каждого слайда указывать: Заголовок, Основной текст, Текст для преподавателя, Интерактив/Вопрос.
- Нумерация: для вставок в существующие файлы — буквенные индексы (№130а, №130б). Для новых блоков допускается сквозная нумерация в пределах дня (пример: М2Д1_№001).
- Тайминг: единая линейка без пересечений и «задвоений» времени; A/B‑слоты отмечать только при пилотных запусках (по отдельному согласованию).
- Рефлексия: в конце каждого блока фиксировать 3–5 «что сделаю завтра» для переноса в лог решений.
- Конфиденциальность: пароли/служебные данные не включать в сценарий; указывать ссылку на техпамятку (закрытый документ аудитории).

### 8. МЕТРИКИ И ПРИВЯЗКА К АОС (без A/B по умолчанию)

- Целевая метрика NPS по АОС: 9.1.
- По каждой сессии фиксировать локальные метрики вовлечённости через итог дня (ниже). Для университетских потоков A/B не применяется по умолчанию из‑за географии и неоднородного качества исполнения.
- Привязка к процессу АОС: использовать SOP и шаблоны из docs/aos/ (SOP, REPORT_TEMPLATE, отчёты волн). По завершении сессии обновлять логи решений (logs/decisions.md) и добавлять инсайты в knowledge/.

Итог дня — микроопрос (2 шкалы + 1 открытый):
- Инструкция: «Оцените по шкале от 0 (совсем нет) до 10 (максимально)».
- В1: «Насколько я был включён в работу этого дня?» (0–10)
- В2: «Насколько темы дня полезны для моей текущей зоны ответственности?» (0–10)
- В3 (открытый): «Что наиболее ценное из изученного сегодня я заберу с собой?»
- Проведение: преподаватель в конце дня просит участников ответить на вопросы, записать на стикерах и прикрепить на доску; по желанию — короткий комментарий выбора.
- Формат сбора: QR‑форма или бумажные стикеры; обязательные поля для цифровой формы: Название дня/модуля, Группа/вуз, Дата.

Приложение (опционально, только пилоты): A/B‑тесты
- Применять только на выделенных пилотных сессиях с подготовкой ведущих и чёткой рандомизацией.
- Сохраняется прежний чек‑лист A/B и шаблон микроопроса pre/post (см. историю изменений версии 2.3).
