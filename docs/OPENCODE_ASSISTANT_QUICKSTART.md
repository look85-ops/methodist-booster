OpenCode Ассистент: Быстрый Старт

Для кого

- Для владельца репозитория: как запустить ассистента в проекте.
- Для ассистента: минимальный протокол работы с инструментами и командными файлами.

1. Подготовка репозитория

- Создайте структуру:
  - `.opencode/agent/` — инструкции суб‑агентов
  - `.opencode/command/` — кастомные команды (`/…`)
  - `docs/` — документация (бэклог, процессы, шаблоны)
  - `logs/decisions.md` — журнал решений
  - `knowledge/` — живая база знаний
- Добавьте `AGENTS.md` с ролями, командами, workflow и правилами (см. этот проект как пример).

2. Роли и команды

- Описывайте каждый суб‑агент в `.opencode/agent/<имя>.md`: роль, зона ответственности, формат ответов.
- Для каждой новой команды создавайте файл в `.opencode/command/` и сразу обновляйте разделы Roles/Commands/Workflow в `AGENTS.md`.

3. Базовые конвенции работы

- Поиск: Glob/Grep. Чтение: Read. Правки: apply_patch. Терминал: Bash для git/gh/npm.
- ASCII по умолчанию. Коммуникация короткая. Минимальные изменения.
- Никогда не откатывать чужие изменения без запроса. Не делать force‑push. Коммитить только по просьбе.

4. Стандартные команды (рекомендуемый набор)

- `/idea` — превращает сырую мысль в фичу → `docs/BACKLOG.md`.
- `/start-task F-NNN` — берёт фичу в работу, проверяет архитектуру.
- `/finish-task` — ритуал обучения (инсайты → `knowledge/`, решение → `logs/decisions.md`).
- `/create-spec` и `/generate-html` — пример сквозного пайплайна ТЗ → артефакт.

5. Git и PR

- Создавайте ветки вида `f/NNN-краткое-название` для фич.
- Коммиты: `[F-NNN] Название фичи`, фокус на «почему».
- PR через `gh pr create`, без force‑push в main. Проверяйте CI.

Шаблон последовательности для PR (PowerShell)

```
# Создать ветку
git checkout -b f/123-quickstart-docs

# Добавить файлы
git add docs/OPENCODE_ASSISTANT_OPERATING_MANUAL.md docs/OPENCODE_ASSISTANT_QUICKSTART.md

# Коммит (по запросу владельца)
git commit -m "[F-123] Добавить операционное руководство и quickstart для ассистента"

# Пуш ветки
git push -u origin f/123-quickstart-docs

# Создать PR с шаблоном тела
gh pr create --title "[F-123] Docs: Operating Manual & Quickstart" --body "$(cat <<'EOF'
## Summary
- Добавлены базовые руководства для ассистента на OpenCode

## Changes
- docs/OPENCODE_ASSISTANT_OPERATING_MANUAL.md
- docs/OPENCODE_ASSISTANT_QUICKSTART.md

## Test Plan
- Ревью контента; проверки ссылок на существующие файлы

## Checklist
- [x] Нет секретов в диффе
- [x] Обновлены связанные разделы
EOF
)"
```

Подробности см. `docs/process/CI_PR_RELEASES.md`.

6. Проверки качества

- Перед «Готово»: минимальные проверки по месту (линтеры/сборка/валидаторы).
- Для документов: непротиворечивость ссылок и обновление связанных разделов.

7. Расширение ассистента

- Новые агенты/команды → сразу обновить Roles/Commands/Workflow в `AGENTS.md`.
- Внешние навыки (skills) фиксируйте в `docs/skills/ADOPTED_SKILLS.md`. Добавляйте команды только если дают пользу.

8. Старт для нового ассистента (чек‑лист)

- Прочитай `AGENTS.md` и `docs/OPENCODE_ASSISTANT_OPERATING_MANUAL.md`.
- Проверь структуру репозитория и наличие базовых папок.
- На первой задаче: определи цель, найди минимальный дифф, спроси при неоднозначности.
- Перед ответом: короткий апдейт → правка → проверка → результат.

Примеры команд

Создать фичу:

```
/idea
Кратко: «[Process] Внести чек‑лист перед релизом в docs/PR_VALIDATION.md»
```

Взять задачу:

```
/start-task F-123
```

Завершить:

```
/finish-task
```

Шаблоны файлов для новых команд и агентов

Новый агент: `.opencode/agent/<agent-name>.md`

```
# <agent-name>

Роль: одна строка.
Зоны ответственности: 3–5 пунктов.
Формат ответов: что обязательно включает.
Guardrails: что не делает.

Workflow:
- Когда вызывается
- Где пишет артефакты
```

Новая команда: `.opencode/command/<command-name>.md`

```
# /<command-name>

Назначение
- Что автоматизирует и когда применять.

Входные данные
- Параметры / аргументы (если есть)

Шаги выполнения
- 1. …
- 2. …

Вывод/артефакты
- Куда и что записывается

Связи
- Какие агенты/файлы затрагиваются
```
