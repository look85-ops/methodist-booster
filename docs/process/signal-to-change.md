# Конвейер «Сигнал → Изменение» (Signal-to-Change)

Цель: системно превращать новые сигналы (данные, фидбэк, сбои, сдвиги рынка) в конкретные изменения в продуктах и процессах с недельным ритмом внедрения.

1. Таксономия сигналов
- Data: метрики и тренды (вовлечённость, ошибки, скорость, стоимость)
- Feedback: отзывы/комментарии пользователей и команды
- Incident: сбои, деградации, регрессы
- Market: конкуренты, цены, регуляторика, партнёры
- Ops: узкие места процесса, долгие циклы, ручные операции
- Hypothesis: внутренние идеи, наводки, «тонкие места»

Атрибуты: категория, источник, сила сигнала (H/M/L), затрагиваемая метрика, срочность, владелец.

2. Статусы и SLA
- new → triaged (до 48ч)
- triaged → in_progress (если принято в работу)
- in_progress → implemented (внесены изменения)
- implemented → measured (замер результата через 3–14 дней)

3. Роутинг (кто что делает)
- Инициатор (любой): фиксирует сигнал по шаблону `templates/signal.md`
- Владелец продукта (NM): приоритизация, назначение владельца изменения
- Исполнитель: вносит изменение и фиксирует запись в `change-log.md`
- Все: ретро раз в неделю (30–45 минут)

4. Недельный ритм (ритуал)
1) Триаж сигналов (10–15 мин): быстро оценить новые, отсеять шум
2) Выбор топ‑1–2 на неделю: максимум одно изменение «побольше» и одно «быстрое»
3) Внедрение и фиксация: короткие PR/правки/скрипты; запись в change‑log
4) Замер и ретро: вердикт «сработало/нет», уроки в `knowledge/process-lessons.md`

5. Метрики процесса
- Доля сигналов, дошедших до изменения, %
- Среднее время от сигнала до изменения, дни
- Доля изменений с положительным эффектом, %

6. Артефакты
- Шаблон сигнала: `docs/process/templates/signal.md`
- Журнал изменений: `docs/process/templates/change-log.md`
- Дашборд статусов: `docs/process/signal-dashboard.md`
- Уроки процесса: `knowledge/process-lessons.md`

Правила
- Минимум бюрократии: коротко, по делу, ссылки на исходники/логи.
- Приватность: не кладём в публичный репозиторий PII/секреты.
